§ 1
System Overview
The backend is a Node.js monolith structured as loosely-coupled modules (game engine, wallet service, auth, notifications). It uses MongoDB as the primary database — chosen for its flexible document model, native JSON support, and ease of horizontal sharding as game volume grows. Redis handles ephemeral state (active game sessions, rate limits, pub/sub for WebSocket fan-out).

🍃 Why MongoDB over PostgreSQL? Game rounds are document-shaped — each round embeds its seeds, revealed cells array, payout, and metadata in one document. MongoDB's flexible schema means no migrations when adding new game types. Time-series collections handle bet history efficiently. Atlas Search provides full-text search on transactions without a separate search cluster.

All routes are prefixed /api/v1. Authenticated routes require Authorization: Bearer <jwt>. Rate limits enforced per user per route via Redis sliding window.

JWT Required

/wallet/deposit/verify

JWT Required

JWT Required

💣 Mines Game

POST
/games/mines/start
Body: {betAmount, mineCount}. Deducts bet atomically from Redis balance. Generates server seed (random 32-byte hex), hashes it (SHA-256), shuffles mine positions. Saves GameRound doc to MongoDB with status="active". Returns {gameId, serverSeedHash, clientSeed, nonce}. Rate limit: 1 active game per user.
JWT Required
POST
/games/mines/reveal
Body: {gameId, cellIndex}. Validates game ownership and status="active". Checks if cellIndex is in minePositions. Updates revealedCells array. Returns {result: "gem"|"mine", currentMultiplier, currentPayout, minePositions (only if mine)}. On mine: sets status="lost", unlocks bet (house keeps). Emits WS event.
JWT Required
POST
/games/mines/cashout
Body: {gameId}. Calculates final payout = betAmount × multiplier(mineCount, revealedCount). Credits payout to Redis balance. Sets status="cashout". Reveals serverSeed for provably fair verification. Creates win transaction record.
JWT Required
GET
/games/mines/verify/:gameId
Returns {serverSeed, clientSeed, nonce, minePositions} for completed rounds. Allows players to independently verify results. Available only after game ends.
JWT Required
GET
/games/history
Query: {page, limit, gameType}. Returns paginated game_rounds sorted by createdAt desc.
JWT Required

§ 6
Authentication Flow
// middleware/validateTelegram.ts
// Telegram sends initData as URL-encoded string.
// We verify it was signed by our bot token.

import { createHmac } from 'crypto'

export function verifyTelegramAuth(initData: string, botToken: string): boolean {
  const params = new URLSearchParams(initData)
  const hash = params.get('hash')
  params.delete('hash')

  // Sort params and build data check string
  const dataCheckString = [...params.entries()]
    .sort(([a], [b]) => a.localeCompare(b))
    .map(([k, v]) => `${k}=${v}`)
    .join('\n')

  // HMAC-SHA256 with key = HMAC-SHA256("WebAppData", botToken)
  const secretKey = createHmac('sha256', 'WebAppData')
    .update(botToken).digest()

  const expectedHash = createHmac('sha256', secretKey)
    .update(dataCheckString).digest('hex')

  return expectedHash === hash
}

§ 8
Withdrawal Queue (BullMQ)

Producer (API Server)

  • User hits POST /wallet/withdraw
  • Balance validated + locked in Redis
  • WithdrawalRequest doc created in MongoDB (status: pending)
  • Job pushed to BullMQ "withdrawals" queue
  • requestId returned to client immediately

Worker (withdrawal.worker.ts)

  • Picks up job from queue (concurrency: 3)
  • Checks reviewRequired flag — if true, pauses until admin approves
  • Calls ton.bridge.ts → sendTransaction to escrow contract
  • Waits for tx confirmation (poll TON API)
  • Updates MongoDB status → "sent", txHash
  • Emits WS event withdrawal:status to user
  • On failure: retries 3× with exponential backoff, then status → "failed"

SecretsVault / EnvBot token, JWT keys, TON private key stored in HashiCorp Vault; injected at runtime

🚨 Critical: The TON private key that signs withdrawals from the escrow contract is stored in an HSM (Hardware Security Module) in production. The backend never holds the raw private key in memory — it calls a signing microservice that has exclusive HSM access. This prevents key extraction even if the app server is fully compromised.
