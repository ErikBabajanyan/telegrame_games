§ 1
Game Overview
Mines is a grid-based game where the player reveals cells one by one, trying to avoid hidden mines. Each successful reveal increases the multiplier. The player can cash out any time before hitting a mine. It's a game of risk management — the player controls when to stop.

How to Play

  • Player selects a bet amount in TON (min 0.01, max 500 TON)
  • Player selects mine count (1, 2, 3, 5, 10, 15, 20, or 24 mines)
  • Player taps "BET" — bet is deducted from balance, 25 cells appear hidden
  • Player taps cells to reveal them — each reveal is either 💎 (safe) or 💣 (mine)
  • Each gem increases the multiplier — the payout shown updates in real time
  • If player hits a mine → round lost, bet forfeited
  • Player can tap "Cash Out" at any time after at least 1 gem → payout credited

§ 3
Provably Fair System
Every game round is provably fair — meaning the player can independently verify that the mine positions were not manipulated after betting. The three-seed system ensures neither the server nor client can cheat.

Three Seeds

  • Server Seed — 32-byte random hex generated per game. NEVER shown until game ends. Prevents server from cheating.
  • Server Seed Hash — SHA-256 of server seed. Shown to player BEFORE game starts. Commits server to the seed.
  • Client Seed — Player-provided (or random). Player can change it before starting. Prevents server from knowing outcome even if they knew client seed.
  • Nonce — Auto-incrementing integer per user. Ensures unique outcomes even with same seeds.

Verification Flow

  • Server generates serverSeed, computes serverSeedHash
  • Player sees serverSeedHash + their clientSeed + nonce BEFORE betting
  • Game plays out (mine positions already determined)
  • After game ends, server reveals raw serverSeed
  • Player can now compute HMAC(serverSeed, "clientSeed:nonce") and derive the exact same mine positions independently
  • Open-source verifier tool published at /verify
// mines.engine.ts — Complete provably fair implementation

import { createHmac, createHash, randomBytes } from 'crypto'

/** Generate cryptographically random server seed */
export function generateServerSeed(): string {
  return randomBytes(32).toString('hex')  // 64-char hex
}

/** Hash server seed to commit to it before game */
export function hashServerSeed(seed: string): string {
  return createHash('sha256').update(seed).digest('hex')
}

/**
 * Deterministically derive mine positions from seeds.
 * Same inputs always produce same outputs — verifiable by anyone.
 */
export function generateMinePositions(
  serverSeed: string,
  clientSeed: string,
  nonce: number,
  mineCount: number
): number[] {
  // 1. Derive combined hash
  const hash = createHmac('sha256', serverSeed)
    .update(`${clientSeed}:${nonce}`)
    .digest('hex')

  // 2. Build array of 25 cells [0..24]
  const cells = Array.from({ length: 25 }, (_, i) => i)

  // 3. Fisher-Yates shuffle using hash bytes as entropy
  for (let i = 24; i > 0; i--) {
    // Take 4 bytes from hash at offset i*4, interpret as uint32
    const offset = (i * 4) % 60
    const val = parseInt(hash.slice(offset, offset + 8), 16)
    const j = val % (i + 1)
    ;[cells[i], cells[j]] = [cells[j], cells[i]]
  }

  // 4. First mineCount positions are the mines
  return cells.slice(0, mineCount).sort((a, b) => a - b)
}

Mathematical Model

Core Formula

Combinations
C(n, k) = n! / (k! × (n−k)!)
Win Probability — survive exactly k reveals with m mines in 25 cells
P(k, m) = C(25−m, k) / C(25, k)
This is the probability that ALL k revealed cells are safe (no mines hit)
Fair Multiplier (no house edge) — after k safe reveals
fair_mult(k, m) = C(25, k) / C(25−m, k) = 1 / P(k, m)
Applied Multiplier — with 2% house edge
mult(k, m) = fair_mult(k, m) × 0.98
RTP = 98%. House edge = 2%. Applied uniformly across all depths and mine counts.
Payout Calculation
payout = bet_amount × mult(k, m)
Next Cell Safety Probability (shown in UI)
P(next_safe | k already revealed) = (25 − m − k) / (25 − k)

UI Elements & Features

Game Controls

  • Bet Amount — numeric input with − / + buttons. Quick-select: ½× and 2× buttons. Min 0.01 TON.
  • Mine Count Selector — horizontal pill buttons: 1, 2, 3, 5, 10, 15, 20, 24
  • BET Button — large CTA, becomes "CASH OUT" after first gem reveal
  • RANDOM Button — auto-picks a random unrevealed safe cell (if possible)
  • Auto Game — toggle: plays with same settings automatically (optional feature)

Live Info Display

  • Next: X.XXx — shows the multiplier the NEXT safe reveal would give
  • Mines: N — mine count reminder
  • Current Payout — real-time: bet × current multiplier in TON
  • serverSeedHash — shown in footer before betting, collapsed by default
  • Balance — top-right, updates instantly after cashout

Cell States & Animations

  • Hidden — dark blue gradient, subtle center dot
  • Tapped (loading) — pulse animation, 150ms
  • Gem revealed — green glow + 💎 bounce-in + haptic
  • Mine hit — red flash + 💣 + board shake + all mines revealed
  • Cashout — all safe cells flip to 💎, confetti burst

Modals & Feedback

  • Game Over Modal — shows bet lost, mine positions, "Play Again" CTA
  • Win Modal — shows payout, profit, new multiplier record
  • Verify Modal — serverSeed + link to verifier after game
  • Insufficient Balance — inline error, link to deposit
  • Haptic feedback — Telegram SDK: light on reveal, heavy on mine, success on cashout

House Risk Model

Expected Value Per Bet

House Expected Value per bet
EV_house = bet × 0.02
Regardless of mine count or how many cells revealed — house always earns 2% in expectation

Variance by Mine Count

Higher mine count = higher variance = house faces more risk from lucky runs, but same long-run EV. The payout cap and bet limits compensate for short-run variance at extreme mine counts.
